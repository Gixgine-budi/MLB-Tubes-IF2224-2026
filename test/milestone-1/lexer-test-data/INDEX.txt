Arion lexer manual test data
Location: MLB-Tubes-IF2224-2026/test/milestone-1/

This folder holds two suites of paired files (*.in.txt + *.expected.txt):

  lexer-test-data/     — 22 numbered happy-path style cases (01–22)
  lexer-broken-inputs/ — error and edge cases (01–21, numbered separately)

Each case is a pair:
  <name>.in.txt        — source input
  <name>.expected.txt  — one logical token per line (see multiline notes below)

Token line format (current ruleset)
  • No space between the token name and the opening parenthesis when a value is
    printed, e.g. ident(Hello), intcon(5), string('TBFO').
  • Bare tokens (no value) are a single word: programsy, semicolon, becomes, …
  • comment(…) lines show only the comment body: text inside { … } or (* … *),
    without the surrounding delimiters. Newlines inside the body are kept, so a
    single comment may span several physical lines in the expected file.
  • unknown(…) lines give the full invalid / unclosed text inside the parentheses
    (including any (* or { that opened the construct), per Milestone 1 error output.
  • string(…) for a string literal always includes the opening and closing single
    quotes of the source literal in the value, e.g. 'abc' → string('abc'), and
    '' → string(''). A pair '' inside the literal stays as two apostrophe
    characters in that value, e.g. 'People''s' → string('People''s').

Gold files (these .expected.txt) follow the official Milestone 1 specification
and the course rules above — not any particular reference implementation. Your
lexer must be updated until its output matches the gold line-for-line (aside
from any tool-specific newline conventions you document).
